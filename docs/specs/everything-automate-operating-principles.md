# Everything Automate 운영 원칙

이 문서는 `everything-automate` 관련 문서와 구현 결정을 다룰 때 공통으로 지켜야 할 원칙을 정리한다.
문서의 목적은 지금 아는 내용을 정리하는 데서 끝나지 않는다.
나중에 누가, 언제, 어디서 읽더라도 이해할 수 있어야 한다.

## 핵심 원칙

### 1. 문서는 시간과 장소에 독립적으로 이해 가능해야 한다

문서는 현재 작업 중인 사람만을 위한 메모가 아니다.
나중에 다른 사람이 이력을 모르고 읽어도 맥락을 따라갈 수 있어야 한다.

- 줄임말이나 내부 약어는 처음 등장할 때 풀어서 쓴다.
- 전제가 있으면 문서 안에 같이 적는다.
- 왜 이 결론에 도달했는지도 함께 남긴다.
- 오늘 기준으로만 맞는 표현은 날짜나 조건을 붙여 제한한다.

### 2. 문서는 현재의 단일 진실원천이 되어야 한다

같은 주제에 서로 다른 설명이 남아 있으면 문서가 아니라 혼란이 된다.
같은 개념은 한 곳에서만 정의하고, 다른 문서는 그 정의를 참조한다.

- 결정된 내용과 검토 중인 내용을 섞지 않는다.
- 오래된 문서는 폐기, 대체, 보류 상태를 분명히 표시한다.
- README 같은 인덱스 문서는 실제 존재하는 문서와 항상 맞아야 한다.

### 3. 문서는 구현 순서와 같은 방향을 따라야 한다

구현이 아직 없는 기능을 문서가 과하게 앞서가면 실행이 꼬인다.
문서는 항상 현재 단계에 맞는 수준에서 써야 한다.

- 먼저 계약을 정의하고, 그 다음 실행 흐름을 정의한다.
- 브로커, 어댑터, 확장 기능은 핵심 계약이 안정된 뒤에 다룬다.
- 한 번에 모든 것을 설명하려 하지 말고, 단계별로 나눈다.

### 4. 문서는 바로 실행 가능한 형태여야 한다

좋은 문서는 읽는 즉시 다음 행동이 보인다.
추상적인 방향만 있고 실제 작업 단위가 없으면 문서로서 불완전하다.

- 각 단계는 산출물, 완료 조건, 다음 단계 의존성을 가져야 한다.
- 구현 문서는 "무엇을 먼저 해야 하는가"를 분명하게 알려야 한다.
- 검증 방법이 있으면 함께 적는다.

### 5. 문서는 변경 이유를 남겨야 한다

무엇을 바꿨는지만 적으면 나중에 판단을 다시 할 수 없다.
왜 그 선택이 더 나은지까지 남겨야 한다.

- 대안이 있었다면 간단히 적는다.
- 버린 방향이 있다면 왜 버렸는지 적는다.
- 재검토 조건이 있으면 명시한다.

### 6. 문서는 구현체보다 오래 살아야 한다

코드는 바뀌어도 문서의 핵심 원칙은 오래 유지되어야 한다.
그래서 문서는 특정 구현 세부사항보다 계약, 의도, 판단 기준을 중심으로 써야 한다.

- 구현 세부는 필요할 때만 적는다.
- 바뀔 가능성이 큰 값은 상수처럼 단정하지 말고 조건으로 적는다.
- 특정 도구 이름보다 역할을 먼저 적는다.

## 작성 규칙

### 용어

- 핵심 용어는 한 문서 안에서 같은 뜻으로만 쓴다.
- 같은 개념에 대해 `계약`, `스키마`, `정의`를 섞어 쓰지 말고 기준을 하나로 맞춘다.

### 구조

- 문서는 짧은 서문, 핵심 규칙, 필요한 경우 예외나 보충 설명 순서로 쓴다.
- 문서 제목만 보고도 역할이 드러나야 한다.
- 표나 목록은 개념을 비교하거나 순서를 보여줄 때만 쓴다.

### 검증

- 새 문서를 만들면 README 인덱스도 같이 갱신한다.
- 실행 순서나 계약을 바꾸면 관련 문서와 충돌하지 않는지 확인한다.
- 변경 후에는 문서가 현재 구현 계획과 어긋나지 않는지 다시 읽어본다.
